Error & failure states

Reference set · all failure modes get a clear recovery path

These are the error states agents and the platform expose. Each one names the specific failure, surfaces the underlying cause without hiding it, and offers a single most-helpful recovery action plus a "show details" affordance for power users.

// Design principles for errors
  • Name the failure honestly — never "something went wrong"
  • State what was preserved — "no changes were made" is the most reassuring sentence in IT
  • One primary recovery action, plus secondary options for power users
  • Show the technical detail — admins are technical and want it visible, not hidden
  • Suggest, don't decide — recommend the next step but let the admin choose
